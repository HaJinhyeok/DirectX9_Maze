## 작업 절차

- 코드 수정은 사용자가 직접 요청했을 때만 수행한다.
- 수정 전 대상 파일과 수정 이유를 먼저 설명한다.
- Roadmap 작업은 한 번에 하나만 `진행 중`으로 둔다.
- 작업 전 해당 항목의 완료 조건과 검증 방법을 확인한다.
- 수정 후 가능한 범위에서 `Debug|x86` 빌드, 검색 또는 실행 결과로 검증한다.
- 성공과 실패를 모두 `docs/WORKLOG.md`에 기록한다.
- 완료 조건을 충족했을 때만 Roadmap 상태를 `완료`로 변경한다.
- 범위 밖에서 발견한 문제는 즉시 수정하지 않고 Roadmap 후보 또는 Worklog에 기록한다.
- 기존 사용자 변경사항과 관련 없는 경고·오류는 이번 작업과 분리해서 보고한다.

## 프로젝트 제약

- 현재 동작을 보존하는 낮은 위험도 정리를 우선한다.
- DirectX 9 및 DirectX SDK June 2010 의존성을 임의로 교체하지 않는다.
- 대규모 클래스 분리, 렌더링 구조 변경, 맵 포맷 변경은 별도 단계로 계획한 뒤 진행한다.
- `docs/TECHNICAL_REVIEW.md`는 로컬 검토 문서이며 공개 커밋 대상에 포함하지 않는다.
- 공개 문서에는 현재 상태와 개선 결과를 과장 없이 기록한다.

## Git 규칙

- 스테이징, 커밋, 브랜치 생성과 원격 작업은 사용자가 명시적으로 요청했을 때만 수행한다.
- 커밋명은 `[HA]`로 시작하고 한글로 작성한다.
- 하나의 Roadmap 항목 또는 하나의 명확한 주제를 한 커밋으로 묶는다.
