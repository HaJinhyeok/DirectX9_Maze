## 클래스 범위 지정 연산자와 함수 선언

### 질문

함수 정의에 `CPlayer::`를 붙이는 경우와 `IsWall()`처럼 붙이지 않는 경우의 차이는 무엇인가?

### 핵심 답변

- `CPlayer::Move()`는 `CPlayer` 클래스의 멤버 함수 정의다.
- 클래스 멤버 함수는 클래스 선언에 포함되어야 하므로 헤더에 선언이 필요하다.
- `IsWall()`은 `CPlayer.cpp`의 익명 namespace에 있는 구현 전용 자유 함수다.
- 구현 파일 안에서만 사용하므로 `CPlayer.h`에 선언할 필요가 없다.

### 프로젝트 적용

벽 조회는 플레이어 공개 인터페이스가 아니라 이동 구현을 돕는 함수이므로 `CPlayer.cpp`의 익명 namespace에 유지했다.

## C++ bool과 Win32 BOOL

### 질문

`BOOL` 대신 일반 `bool`을 사용해도 되는가?

### 핵심 답변

- `bool`은 C++ 기본 타입이며 값은 `true`, `false`다.
- `BOOL`은 Win32가 정의한 정수 타입이며 API 호환을 위해 `TRUE`, `FALSE`와 함께 사용된다.
- 프로젝트 내부 로직에는 `bool`, Win32 API 경계에는 API가 요구하는 `BOOL`을 사용하는 것이 명확하다.

### 프로젝트 적용

내부 구현 전용 `IsWall()`은 `bool`을 사용한다. 기존 Win32·DirectX 시그니처와 맞닿은 코드는 단계적으로 판단한다.

## SafeRelease 템플릿과 noexcept

### 질문

`SafeRelease()`의 `Release()`가 IntelliSense에서 unknown으로 보이는 이유와 `noexcept`의 의미는 무엇인가?

### 핵심 답변

- `SafeRelease()`는 템플릿이라 `T`가 실제 COM 포인터 타입으로 결정된 뒤 `Release()` 호출이 유효한지 컴파일러가 검사한다.
- 편집 중 IntelliSense가 구체적인 `T`를 알지 못하면 `Release()`를 unknown으로 표시할 수 있지만 실제 빌드 결과가 최종 판단 기준이다.
- `noexcept`는 함수가 예외를 밖으로 전달하지 않는다는 계약이다.
- COM `Release()`와 포인터의 `nullptr` 대입만 수행하는 정리 함수에 적합하다.

### 프로젝트 적용

COM 포인터 해제는 `SafeRelease()`로 통일했고 `Debug|x86` 빌드로 템플릿 인스턴스가 유효함을 확인했다.

## 프로세스 종료와 윈도우 종료 메시지

### 질문

`exit(0)`도 게임을 종료하는데 `PostMessage(hWnd, WM_CLOSE, 0, 0)`로 바꾸면 무엇이 다른가?

### 핵심 답변

- `exit(0)`은 프로세스 종료를 시작하므로 프로젝트의 `WM_DESTROY` 처리와 `CleanUp()`을 거치지 않는다.
- `WM_CLOSE` 메시지는 기본 윈도우 종료 흐름을 통해 `WM_DESTROY`, `CleanUp()`, `PostQuitMessage(0)` 순서로 이어진다.
- 운영체제가 프로세스 자원을 회수하므로 화면상 결과는 같아 보이지만 명시적인 정리 경로를 보장한다는 차이가 있다.

### 프로젝트 적용

게임 내부 종료 버튼도 타이틀 바 종료와 같은 메시지 기반 정리 경로를 사용하도록 통일했다.

## 조명 상태에 따른 FPS 차이

### 질문

밤에는 약 120 FPS, 낮에는 약 400~500 FPS가 나오는 것이 정상인가?

### 핵심 답변

- 밤에는 `D3DRS_LIGHTING`이 활성화되어 정점 조명 계산이 수행된다.
- 장치가 `D3DCREATE_SOFTWARE_VERTEXPROCESSING`으로 생성되어 해당 계산 비용이 CPU에 집중된다.
- 낮에는 조명 계산이 비활성화되고 수직 동기화도 없어 가능한 만큼 렌더링하므로 FPS가 크게 높아질 수 있다.
- 오류로 단정할 현상은 아니지만 조명 경로가 성능 병목이라는 측정 신호다.

### 프로젝트 적용

동일 장면의 조명 상태별 frame time과 정점 처리 방식을 비교하는 작업을 `P3-08`로 등록했다.

## 매크로 상수와 constexpr

### 질문

매크로 상수를 `constexpr`로 바꾸면 왜 더 좋은가?

### 핵심 답변

- 매크로는 전처리기의 문자열 치환이라 타입과 C++ 유효 범위가 없다.
- `constexpr`는 타입, 범위와 컴파일 시간 상수 보장을 제공한다.
- IDE의 정의 찾기, 참조 검색, 이름 변경과 디버깅에도 더 유리하다.
- 조건부 컴파일, 토큰 결합 또는 외부 API 관례처럼 전처리가 필요한 경우에는 매크로를 유지한다.

### 프로젝트 적용

일반 수치·문자열·색상은 `constexpr`, DirectX FVF 조합은 매크로로 분류한다.

## 상수의 k 접두사

### 질문

`kWindowWidth`의 `k`는 어떤 의미인가?

### 핵심 답변

- `k`는 constant를 나타내는 네이밍 표식이며 C++ 문법은 아니다.
- `constexpr`는 컴파일러가 상수임을 보장하고 `k`는 사람이 이름만 보고 상수임을 알아보게 한다.
- 일반 변수의 `camelCase`와 컴파일 시간 상수의 `kPascalCase`를 구분한다.

### 프로젝트 적용

프로젝트의 컴파일 시간 상수는 `kPascalCase` 규칙을 사용한다.

## 계산식과 constexpr

### 질문

`SQRT2`나 `ROTATION_AMOUNT`처럼 계산식으로 정의된 매크로도 `constexpr`로 바꿀 수 있는가?

### 핵심 답변

- 컴파일 시간에 평가할 수 있는 피연산자와 연산만 사용한다면 계산식도 `constexpr`로 정의할 수 있다.
- `D3DX_PI / 200.0f` 같은 산술식은 `constexpr float`로 옮길 수 있다.
- 현재 프로젝트의 `sqrtf(2.0f)`는 사용 중인 C++ 표준에서 `constexpr` 함수로 보장되지 않으므로, 제곱근 2의 값을 명시적인 부동소수점 상수로 정의한다.
- 매크로 정의만 바꾸면 기존 이름을 사용하는 코드가 컴파일되지 않으므로 선언과 모든 사용처를 함께 변경해야 한다.

### 프로젝트 적용

`SQRT2`는 `kSqrt2`, `ROTATION_AMOUNT`는 `kRotationAmount`로 전환하고, 이전 이름이 남지 않았는지 검색한 뒤 전체 빌드로 검증한다.

## 오버라이딩 함수에서 기반 구현 호출

### 질문

기반 클래스와 파생 클래스의 `Initialize()`를 `virtual`과 `override`로 선언하면 파생 함수 내부의 재귀 호출을 해결할 수 있는가?

### 핵심 답변

- `virtual`은 기반 클래스 포인터나 참조를 통한 호출이 실제 객체의 재정의 함수로 연결되게 한다.
- 파생 클래스의 `Initialize()` 안에서 한정 없이 `Initialize(position)`을 호출하면 현재 파생 함수가 다시 호출되므로 무한 재귀가 발생한다.
- 기반 구현을 재사용하려면 가상 함수 여부와 관계없이 `CNotice::Initialize(position)`처럼 클래스 범위를 명시해야 한다.
- `override`는 재정의 의도가 있을 때 컴파일러가 시그니처 일치를 검사하게 하지만 기반 구현을 자동 호출하지는 않는다.

### 프로젝트 적용

현재 호출 경로에는 런타임 다형성이 필요하지 않으므로 `virtual`을 추가하지 않고 `CExit::Initialize()`에서 `CNotice::Initialize(position)`을 명시적으로 호출한다.

## 프러스텀의 구 교차 판정

### 질문

점과 평면 사이의 거리를 계산하는 함수 이름에 왜 `Sphere`가 들어가는가?

### 핵심 답변

- 함수는 전달된 위치와 각 프러스텀 평면 사이의 부호 있는 거리를 계산한다.
- 위치를 구의 중심, 두 번째 거리 값을 반지름으로 보고 `평면 거리 < -반지름`이면 구가 해당 평면 바깥에 완전히 놓였다고 판정한다.
- 어느 평면에서도 완전히 배제되지 않으면 구가 프러스텀 내부에 있거나 경계와 교차하므로 `IntersectsSphere`가 실제 반환 의미에 가깝다.
- 구가 완전히 내부에 있는지만 검사하는 `ContainsSphere`와는 판정 조건이 다르다.
- 반지름을 `0`으로 전달하면 같은 식을 점 포함 판정으로 사용할 수 있다.

### 프로젝트 적용

`CFrustum::IsInFrustum(position, distance)`를 `IntersectsSphere(center, radius)` 의미로 해석하고, 함수 이름은 `IntersectsSphere`로 변경한다. 매개변수 이름 통일은 별도 단계에서 진행한다.

## 대표 타입이 없는 자유 함수 모듈의 파일명

### 질문

`main.cpp`처럼 관례적인 진입점이 아닌 `maze_function.h/.cpp` 같은 자유 함수 모듈은 현재 파일명 규칙의 예외로 두어도 되는가?

### 핵심 답변

- 대표 클래스가 없는 파일도 이름만 보고 책임을 파악할 수 있어야 한다.
- `function`, `util`, `helper`처럼 범위가 넓은 이름보다 모듈이 다루는 도메인이나 역할을 사용한다.
- 프로젝트 파일 규칙이 `PascalCase`라면 자유 함수 모듈도 같은 표기법을 적용한다.
- 언어나 도구가 요구하는 `main.cpp` 같은 관례적 진입점은 예외로 유지할 수 있다.

### 프로젝트 적용

미로 벽, 표지판과 출구 생성을 담당하는 `maze_function.h/.cpp`를 `MazeGenerator.h/.cpp`로 변경한다.

## private 멤버 변수의 `m_` 접두사

### 질문

private 멤버 변수를 `m_` + `camelCase`로 작성하는 방식은 실제로 널리 사용되고 선호되는가?

### 핵심 답변

- `m_memberName`은 C++에서 널리 쓰이는 선택지 중 하나지만 언어 전체의 단일 표준이나 압도적인 선호 방식은 아니다.
- 대표적인 대안으로 접미사를 붙이는 `member_name_`, 별도 표식 없는 `memberName` 또는 `MemberName`이 있다.
- `m_`는 자료형이 아니라 멤버라는 범위를 표시하므로 `p`, `dw` 같은 타입 기반 헝가리안 표기와 목적이 다르다.
- 장점은 매개변수와 지역 변수에서 멤버를 즉시 구분하고 이름 충돌을 줄이는 것이며, 단점은 접두사로 인한 시각적 잡음이다.
- 어떤 형식을 택하든 프로젝트 전체 일관성과 도구 자동 검사가 특정 형식 자체보다 중요하다.

### 프로젝트 적용

기존 코드에 `m_` 계열 이름이 많고 전역 변수에는 `g_` 범위 표식을 사용하므로, private 멤버는 `m_` + `camelCase`로 통일한다.

## 값 객체와 댕글링 포인터

### 질문

값 객체가 포인터보다 적절하다는 것은 객체 자체를 직접 보관하는 편이 낫다는 뜻인가? 댕글링 포인터는 무엇인가?

### 핵심 답변

- 값 객체는 객체 자체를 변수에 보관하며 변수가 유효한 범위를 벗어날 때 소멸자가 자동으로 호출된다.
- 객체의 수명이 소유자의 수명과 같고 선택적 존재, 다형성 또는 독립적인 동적 수명이 필요하지 않다면 값 객체를 우선 고려한다.
- 포인터가 필요한 상황도 있으므로 포인터 자체가 나쁜 것은 아니다. 중요한 것은 소유자와 수명 규칙이 명확한지 여부다.
- 댕글링 포인터는 이미 소멸되거나 해제된 객체의 주소를 계속 가지고 있는 포인터다.
- 댕글링 포인터를 역참조하면 정의되지 않은 동작이 발생하며 충돌, 메모리 훼손 또는 간헐적인 오류로 이어질 수 있다.
- 동적 소유가 필요하다면 직접 `new`와 `delete`를 관리하기보다 `std::unique_ptr` 같은 RAII 소유 타입을 우선 고려한다.

### 프로젝트 적용

`main.cpp`의 세 `POINT`와 `Frustum`은 프로그램 전체와 수명이 같고 다형성이나 선택적 생성도 필요하지 않다. 따라서 `new`로 생성한 포인터 대신 파일 범위 값 객체로 보관해 수동 `delete`와 댕글링 포인터 가능성을 제거한다.

## 코드 주석의 유지 기준

### 질문

필드나 함수 이름으로 어느 정도 드러나는 내용이라도 Direct3D 정점 형식이나 생성 대상의 의미를 설명하기 위해 주석을 유지해도 되는가?

### 핵심 답변

- 주석은 코드가 무엇을 하는지 그대로 번역하기보다 해당 코드가 필요한 이유, 제약과 도메인 의미를 보충할 때 가치가 크다.
- 학습용 또는 저수준 API 코드에서는 API 개념과 실제 데이터의 대응 관계를 보여주는 주석도 유용하다.
- 이름만 반복하는 주석은 삭제하되, 목적이나 사용 맥락을 추가할 수 있다면 구체적으로 고쳐서 유지한다.
- 주석이 현재 코드와 달라지면 잘못된 정보를 제공하므로 이름 변경과 구조 변경 시 함께 검토해야 한다.

### 프로젝트 적용

타일 정점의 `normal`, `position`, `textureCoordinate` 구간은 각각 `D3DFVF_NORMAL`, `D3DFVF_XYZ`, `D3DFVF_TEX1`과의 대응 및 렌더링 목적이 드러나도록 주석을 유지한다. 기본 메시 생성 주석도 단순 도형 종류보다 플레이어·총알 표시와 스카이박스 렌더링이라는 용도를 설명하도록 작성한다.

## 파일 범위 함수의 `static`

### 질문

`main.cpp`에서만 사용하는 자유 함수 앞에 `static`을 붙이는 이유는 무엇인가?

### 핵심 답변

- 네임스페이스 범위의 자유 함수에 붙은 `static`은 해당 함수에 내부 연결을 부여해 현재 번역 단위에서만 이름을 사용할 수 있게 한다.
- 다른 `.cpp` 파일에 같은 이름의 함수가 있어도 링커 심벌이 충돌하지 않으며, 함수가 파일 밖의 공개 인터페이스가 아니라는 의도를 나타낸다.
- 함수 내부 지역 변수의 `static`이나 클래스의 정적 멤버와는 의미가 다르다.
- 내부 연결이 반드시 필요한 것은 아니며, 현대 C++에서는 익명 네임스페이스도 같은 목적으로 널리 사용한다.
- 나중에 다른 파일에서 호출해야 한다면 `static`을 제거하고 헤더 선언 또는 명시적인 클래스·모듈 인터페이스를 제공해야 한다.

### 프로젝트 적용

`main.cpp`의 초기화, 입력 처리, 갱신, 렌더링과 해제 보조 함수는 현재 다른 번역 단위에서 호출하지 않으므로 `static`으로 파일 내부 소유를 표시한다. 운영체제가 진입점으로 찾는 `WinMain()`은 외부 연결을 유지한다.

## 리소스 소유 클래스의 특수 멤버 함수

### 질문

`XFileModel` 헤더의 생성자, 소멸자와 `= delete`로 선언한 복사 함수는 각각 어떤 역할을 하는가?

### 핵심 답변

- `XFileModel();`은 기본 생성자 선언이며 새 객체의 초기 상태를 만든다. 실제 구현에서는 리소스 포인터를 null, 개수를 0으로 초기화한다.
- `~XFileModel();`은 소멸자 선언이며 객체 수명이 끝날 때 소유한 COM 리소스와 동적 배열을 해제한다.
- `XFileModel(const XFileModel&) = delete;`는 복사 생성을 금지한다.
- `XFileModel& operator=(const XFileModel&) = delete;`는 기존 객체에 대한 복사 대입을 금지한다.
- 리소스 포인터를 단순 복사하면 두 객체가 같은 리소스를 소유한다고 오해해 이중 해제할 수 있으므로 복사를 막는다.
- 생성자와 소멸자의 헤더 표기는 선언이고 `.cpp`의 본문이 정의다. `= delete`가 붙은 함수는 사용할 수 없도록 삭제된 함수로 정의한 것이다.

### 프로젝트 적용

`XFileModel`은 메시, 재질 배열과 텍스처 배열을 직접 소유하므로 복사할 수 없는 값 타입으로 시작한다. 추후 이동이 필요해지기 전까지는 하나의 `Tiger` 내부에서 생성되고 소멸되는 수명만 허용한다.

## 미구현 기능을 위한 빈 확장 지점

### 질문

나중에 점프나 추가 기능을 구현할 가능성에 대비해 현재 동작이 없는 스페이스바 입력을 별도 함수로 유지하는 것이 좋은가?

### 핵심 답변

- 빈 조건문이나 빈 함수는 실제 요구사항과 책임이 정해지기 전에 만든 추측성 확장 지점이다.
- 확장 가능성만으로 코드를 미리 만들면 호출 경로와 죽은 코드가 늘고, 이름이 미래 기능의 실제 구조와 맞지 않을 수 있다.
- 플러그인 콜백이나 외부 인터페이스처럼 미리 고정해야 하는 계약이 아니라면 기능을 구현할 때 필요한 경계를 만드는 편이 낫다.
- 미래 기능은 Roadmap이나 TODO 문서에 후보와 설계 영향을 기록하고, 코드에는 현재 동작에 필요한 경로만 유지한다.
- 점프는 입력뿐 아니라 Y축 이동, 중력, 바닥 판정, 충돌과 카메라 처리가 필요하므로 빈 입력 함수만으로 적절한 확장 구조를 결정할 수 없다.

### 프로젝트 적용

일반적으로는 빈 확장 지점을 만들지 않는 것을 기본으로 한다. 다만 이 프로젝트에서는 사용자가 스페이스바 기능 확장 의도를 명시했으므로 `HandleJumpInput()`과 빈 `Player::Jump()`를 의도적인 확장 지점으로 유지한다. 실제 점프 또는 보조 액션 구현은 입력 외의 이동·충돌 요구사항을 정한 뒤 별도 작업으로 진행한다.

## Win32 비차단 메시지 루프

### 질문

`PeekMessage()`, `TranslateMessage()`와 `DispatchMessage()`로 구성된 메인 루프는 어떤 역할을 하는가?

### 핵심 답변

- `PeekMessage()`는 현재 스레드의 메시지 큐를 확인하며 메시지가 없어도 기다리지 않고 즉시 반환한다.
- 첫 번째 `NULL`은 특정 창이 아니라 현재 스레드에 속한 모든 창의 메시지를 대상으로 한다.
- `0U, 0U`는 메시지 범위를 제한하지 않는다는 뜻이고 `PM_REMOVE`는 읽은 메시지를 큐에서 제거한다.
- `TranslateMessage()`는 `WM_KEYDOWN` 같은 키 입력을 문자 입력인 `WM_CHAR`로 변환할 수 있도록 보조한다.
- `DispatchMessage()`는 메시지를 해당 창의 윈도우 프로시저인 `MsgProc()`로 전달한다.
- 메시지가 없을 때 `else`에서 게임 갱신과 렌더링을 실행하므로 창이 응답하면서도 게임 루프가 계속 동작한다.

### 프로젝트 적용

메시지가 있는 프레임에는 메시지를 하나 처리하고, 큐가 비었을 때 `UpdateInput()` → `UpdateGame()` → `Render()`를 실행한다. 현재 delta time은 메시지를 처리한 시간까지 포함한 실제 경과 시간으로 계산된다. 메시지가 계속 쌓이면 게임 갱신이 늦어질 수 있으므로 향후 필요하면 한 프레임에 메시지를 모두 비우는 구조를 검토한다.

## delta time 기반 키보드 회전과 이벤트 기반 마우스 회전

### 질문

키보드 회전에 delta time을 전달하도록 바꿀 때 `MsgProc()`에서 호출하는 마우스 회전은 어떻게 처리해야 하는가?

### 핵심 답변

- 키보드 회전은 키를 누르는 동안 매 프레임 실행되므로 `초당 각속도 × delta time`으로 회전량을 계산한다.
- 마우스 회전은 `WM_MOUSEMOVE`가 전달한 커서 이동 픽셀을 감도와 곱해 이번 이벤트의 회전량을 계산한다.
- 마우스 이동량 자체가 이벤트 사이의 변화를 나타내므로 같은 값에 delta time을 다시 곱하면 프레임률에 따라 감도가 달라질 수 있다.
- C++ 함수 오버로딩을 사용하면 키보드용 `Rotate(BOOL, FLOAT)`와 마우스용 `Rotate(BOOL, BOOL, FLOAT)`를 인자 개수로 구분할 수 있다.
- 마우스 상하 회전 제한은 고정 키보드 회전량이 아니라 실제 전달된 `angle`을 기준으로 검사해야 한다.

### 프로젝트 적용

`HandleRotationInput()`은 키보드 회전에 `deltaTimeSeconds`를 전달한다. `MsgProc()`의 세 인자 `Rotate()` 호출은 픽셀 기반 `angle`을 그대로 전달하며 delta time을 추가하지 않는다. 갱신되지 않아 의미가 없던 `dwRotationTime` 기반 제한은 제거한다.

## private 멤버 변수의 `m_` 접두사

### 질문

private 멤버 변수를 `m_` + `camelCase`로 작성하는 방식은 실제로 널리 사용되고 선호되는가?

### 핵심 답변

- `m_memberName`은 C++에서 널리 쓰이는 선택지 중 하나지만 언어 전체의 단일 표준이나 압도적인 선호 방식은 아니다.
- 대표적인 대안으로 접미사를 붙이는 `member_name_`, 별도 표식 없는 `memberName` 또는 `MemberName`이 있다.
- `m_`는 자료형이 아니라 멤버라는 범위를 표시하므로 `p`, `dw` 같은 타입 기반 헝가리안 표기와 목적이 다르다.
- 장점은 매개변수와 지역 변수에서 멤버를 즉시 구분하고 이름 충돌을 줄이는 것이며, 단점은 접두사로 인한 시각적 잡음이다.
- 어떤 형식을 택하든 프로젝트 전체 일관성과 도구 자동 검사가 특정 형식 자체보다 중요하다.

### 프로젝트 적용

기존 코드에 `m_` 계열 이름이 많고 전역 변수에는 `g_` 범위 표식을 사용하므로, private 멤버는 `m_` + `camelCase`로 통일한다.
