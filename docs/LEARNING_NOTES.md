## 클래스 범위 지정 연산자와 함수 선언

### 질문

함수 정의에 `CPlayer::`를 붙이는 경우와 `IsWall()`처럼 붙이지 않는 경우의 차이는 무엇인가?

### 핵심 답변

- `CPlayer::Move()`는 `CPlayer` 클래스의 멤버 함수 정의다.
- 클래스 멤버 함수는 클래스 선언에 포함되어야 하므로 헤더에 선언이 필요하다.
- `IsWall()`은 `CPlayer.cpp`의 익명 namespace에 있는 구현 전용 자유 함수다.
- 구현 파일 안에서만 사용하므로 `CPlayer.h`에 선언할 필요가 없다.

### 프로젝트 적용

벽 조회는 플레이어 공개 인터페이스가 아니라 이동 구현을 돕는 함수이므로 `CPlayer.cpp`의 익명 namespace에 유지했다.

## C++ bool과 Win32 BOOL

### 질문

`BOOL` 대신 일반 `bool`을 사용해도 되는가?

### 핵심 답변

- `bool`은 C++ 기본 타입이며 값은 `true`, `false`다.
- `BOOL`은 Win32가 정의한 정수 타입이며 API 호환을 위해 `TRUE`, `FALSE`와 함께 사용된다.
- 프로젝트 내부 로직에는 `bool`, Win32 API 경계에는 API가 요구하는 `BOOL`을 사용하는 것이 명확하다.

### 프로젝트 적용

내부 구현 전용 `IsWall()`은 `bool`을 사용한다. 기존 Win32·DirectX 시그니처와 맞닿은 코드는 단계적으로 판단한다.

## SafeRelease 템플릿과 noexcept

### 질문

`SafeRelease()`의 `Release()`가 IntelliSense에서 unknown으로 보이는 이유와 `noexcept`의 의미는 무엇인가?

### 핵심 답변

- `SafeRelease()`는 템플릿이라 `T`가 실제 COM 포인터 타입으로 결정된 뒤 `Release()` 호출이 유효한지 컴파일러가 검사한다.
- 편집 중 IntelliSense가 구체적인 `T`를 알지 못하면 `Release()`를 unknown으로 표시할 수 있지만 실제 빌드 결과가 최종 판단 기준이다.
- `noexcept`는 함수가 예외를 밖으로 전달하지 않는다는 계약이다.
- COM `Release()`와 포인터의 `nullptr` 대입만 수행하는 정리 함수에 적합하다.

### 프로젝트 적용

COM 포인터 해제는 `SafeRelease()`로 통일했고 `Debug|x86` 빌드로 템플릿 인스턴스가 유효함을 확인했다.

## 프로세스 종료와 윈도우 종료 메시지

### 질문

`exit(0)`도 게임을 종료하는데 `PostMessage(hWnd, WM_CLOSE, 0, 0)`로 바꾸면 무엇이 다른가?

### 핵심 답변

- `exit(0)`은 프로세스 종료를 시작하므로 프로젝트의 `WM_DESTROY` 처리와 `CleanUp()`을 거치지 않는다.
- `WM_CLOSE` 메시지는 기본 윈도우 종료 흐름을 통해 `WM_DESTROY`, `CleanUp()`, `PostQuitMessage(0)` 순서로 이어진다.
- 운영체제가 프로세스 자원을 회수하므로 화면상 결과는 같아 보이지만 명시적인 정리 경로를 보장한다는 차이가 있다.

### 프로젝트 적용

게임 내부 종료 버튼도 타이틀 바 종료와 같은 메시지 기반 정리 경로를 사용하도록 통일했다.

## 조명 상태에 따른 FPS 차이

### 질문

밤에는 약 120 FPS, 낮에는 약 400~500 FPS가 나오는 것이 정상인가?

### 핵심 답변

- 밤에는 `D3DRS_LIGHTING`이 활성화되어 정점 조명 계산이 수행된다.
- 장치가 `D3DCREATE_SOFTWARE_VERTEXPROCESSING`으로 생성되어 해당 계산 비용이 CPU에 집중된다.
- 낮에는 조명 계산이 비활성화되고 수직 동기화도 없어 가능한 만큼 렌더링하므로 FPS가 크게 높아질 수 있다.
- 오류로 단정할 현상은 아니지만 조명 경로가 성능 병목이라는 측정 신호다.

### 프로젝트 적용

동일 장면의 조명 상태별 frame time과 정점 처리 방식을 비교하는 작업을 `P3-08`로 등록했다.

## 매크로 상수와 constexpr

### 질문

매크로 상수를 `constexpr`로 바꾸면 왜 더 좋은가?

### 핵심 답변

- 매크로는 전처리기의 문자열 치환이라 타입과 C++ 유효 범위가 없다.
- `constexpr`는 타입, 범위와 컴파일 시간 상수 보장을 제공한다.
- IDE의 정의 찾기, 참조 검색, 이름 변경과 디버깅에도 더 유리하다.
- 조건부 컴파일, 토큰 결합 또는 외부 API 관례처럼 전처리가 필요한 경우에는 매크로를 유지한다.

### 프로젝트 적용

일반 수치·문자열·색상은 `constexpr`, DirectX FVF 조합은 매크로로 분류한다.

## 상수의 k 접두사

### 질문

`kWindowWidth`의 `k`는 어떤 의미인가?

### 핵심 답변

- `k`는 constant를 나타내는 네이밍 표식이며 C++ 문법은 아니다.
- `constexpr`는 컴파일러가 상수임을 보장하고 `k`는 사람이 이름만 보고 상수임을 알아보게 한다.
- 일반 변수의 `camelCase`와 컴파일 시간 상수의 `kPascalCase`를 구분한다.

### 프로젝트 적용

프로젝트의 컴파일 시간 상수는 `kPascalCase` 규칙을 사용한다.
