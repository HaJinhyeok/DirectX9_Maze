# Learning Notes

프로젝트 작업 중 질문한 C++·Win32·DirectX 개념과 설계 의도를 요약한다. 변경 이력은 `WORKLOG.md`, 확정된 규칙은 `CODING_CONVENTIONS.md`에 기록하고 이 문서에는 이해와 판단에 도움이 되는 내용을 남긴다.

## 기록 기준

- 용어의 의미, 두 방식의 차이, 선택 이유와 트레이드오프를 기록한다.
- 채팅 전문을 복사하지 않고 질문, 핵심 답변과 프로젝트 적용만 요약한다.
- 일시적인 진행 상황과 단순 빌드 결과는 `WORKLOG.md`에만 기록한다.
- 비밀번호, 토큰, 개인 정보와 공개할 수 없는 코드는 기록하지 않는다.

## 클래스 범위 지정 연산자와 함수 선언

### 질문

함수 정의에 `CPlayer::`를 붙이는 경우와 `IsWall()`처럼 붙이지 않는 경우의 차이는 무엇인가?

### 핵심 답변

- `CPlayer::Move()`는 `CPlayer` 클래스의 멤버 함수 정의다.
- 클래스 멤버 함수는 클래스 선언에 포함되어야 하므로 헤더에 선언이 필요하다.
- `IsWall()`은 `CPlayer.cpp`의 익명 namespace에 있는 구현 전용 자유 함수다.
- 구현 파일 안에서만 사용하므로 `CPlayer.h`에 선언할 필요가 없다.

### 프로젝트 적용

벽 조회는 플레이어 공개 인터페이스가 아니라 이동 구현을 돕는 함수이므로 `CPlayer.cpp`의 익명 namespace에 유지했다.

## C++ bool과 Win32 BOOL

### 질문

`BOOL` 대신 일반 `bool`을 사용해도 되는가?

### 핵심 답변

- `bool`은 C++ 기본 타입이며 값은 `true`, `false`다.
- `BOOL`은 Win32가 정의한 정수 타입이며 API 호환을 위해 `TRUE`, `FALSE`와 함께 사용된다.
- 프로젝트 내부 로직에는 `bool`, Win32 API 경계에는 API가 요구하는 `BOOL`을 사용하는 것이 명확하다.

### 프로젝트 적용

내부 구현 전용 `IsWall()`은 `bool`을 사용한다. 기존 Win32·DirectX 시그니처와 맞닿은 코드는 단계적으로 판단한다.

## SafeRelease 템플릿과 noexcept

### 질문

`SafeRelease()`의 `Release()`가 IntelliSense에서 unknown으로 보이는 이유와 `noexcept`의 의미는 무엇인가?

### 핵심 답변

- `SafeRelease()`는 템플릿이라 `T`가 실제 COM 포인터 타입으로 결정된 뒤 `Release()` 호출이 유효한지 컴파일러가 검사한다.
- 편집 중 IntelliSense가 구체적인 `T`를 알지 못하면 `Release()`를 unknown으로 표시할 수 있지만 실제 빌드 결과가 최종 판단 기준이다.
- `noexcept`는 함수가 예외를 밖으로 전달하지 않는다는 계약이다.
- COM `Release()`와 포인터의 `nullptr` 대입만 수행하는 정리 함수에 적합하다.

### 프로젝트 적용

COM 포인터 해제는 `SafeRelease()`로 통일했고 `Debug|x86` 빌드로 템플릿 인스턴스가 유효함을 확인했다.

## 프로세스 종료와 윈도우 종료 메시지

### 질문

`exit(0)`도 게임을 종료하는데 `PostMessage(hWnd, WM_CLOSE, 0, 0)`로 바꾸면 무엇이 다른가?

### 핵심 답변

- `exit(0)`은 프로세스 종료를 시작하므로 프로젝트의 `WM_DESTROY` 처리와 `CleanUp()`을 거치지 않는다.
- `WM_CLOSE` 메시지는 기본 윈도우 종료 흐름을 통해 `WM_DESTROY`, `CleanUp()`, `PostQuitMessage(0)` 순서로 이어진다.
- 운영체제가 프로세스 자원을 회수하므로 화면상 결과는 같아 보이지만 명시적인 정리 경로를 보장한다는 차이가 있다.

### 프로젝트 적용

게임 내부 종료 버튼도 타이틀 바 종료와 같은 메시지 기반 정리 경로를 사용하도록 통일했다.

## 조명 상태에 따른 FPS 차이

### 질문

밤에는 약 120 FPS, 낮에는 약 400~500 FPS가 나오는 것이 정상인가?

### 핵심 답변

- 밤에는 `D3DRS_LIGHTING`이 활성화되어 정점 조명 계산이 수행된다.
- 장치가 `D3DCREATE_SOFTWARE_VERTEXPROCESSING`으로 생성되어 해당 계산 비용이 CPU에 집중된다.
- 낮에는 조명 계산이 비활성화되고 수직 동기화도 없어 가능한 만큼 렌더링하므로 FPS가 크게 높아질 수 있다.
- 오류로 단정할 현상은 아니지만 조명 경로가 성능 병목이라는 측정 신호다.

### 프로젝트 적용

동일 장면의 조명 상태별 frame time과 정점 처리 방식을 비교하는 작업을 `P3-08`로 등록했다.

## 매크로 상수와 constexpr

### 질문

매크로 상수를 `constexpr`로 바꾸면 왜 더 좋은가?

### 핵심 답변

- 매크로는 전처리기의 문자열 치환이라 타입과 C++ 유효 범위가 없다.
- `constexpr`는 타입, 범위와 컴파일 시간 상수 보장을 제공한다.
- IDE의 정의 찾기, 참조 검색, 이름 변경과 디버깅에도 더 유리하다.
- 조건부 컴파일, 토큰 결합 또는 외부 API 관례처럼 전처리가 필요한 경우에는 매크로를 유지한다.

### 프로젝트 적용

일반 수치·문자열·색상은 `constexpr`, DirectX FVF 조합은 매크로로 분류한다.

## 상수의 k 접두사

### 질문

`kWindowWidth`의 `k`는 어떤 의미인가?

### 핵심 답변

- `k`는 constant를 나타내는 네이밍 표식이며 C++ 문법은 아니다.
- `constexpr`는 컴파일러가 상수임을 보장하고 `k`는 사람이 이름만 보고 상수임을 알아보게 한다.
- 일반 변수의 `camelCase`와 컴파일 시간 상수의 `kPascalCase`를 구분한다.

### 프로젝트 적용

프로젝트의 컴파일 시간 상수는 `kPascalCase` 규칙을 사용한다.

## 계산식과 constexpr

### 질문

`SQRT2`나 `ROTATION_AMOUNT`처럼 계산식으로 정의된 매크로도 `constexpr`로 바꿀 수 있는가?

### 핵심 답변

- 컴파일 시간에 평가할 수 있는 피연산자와 연산만 사용한다면 계산식도 `constexpr`로 정의할 수 있다.
- `D3DX_PI / 200.0f` 같은 산술식은 `constexpr float`로 옮길 수 있다.
- 현재 프로젝트의 `sqrtf(2.0f)`는 사용 중인 C++ 표준에서 `constexpr` 함수로 보장되지 않으므로, 제곱근 2의 값을 명시적인 부동소수점 상수로 정의한다.
- 매크로 정의만 바꾸면 기존 이름을 사용하는 코드가 컴파일되지 않으므로 선언과 모든 사용처를 함께 변경해야 한다.

### 프로젝트 적용

`SQRT2`는 `kSqrt2`, `ROTATION_AMOUNT`는 `kRotationAmount`로 전환하고, 이전 이름이 남지 않았는지 검색한 뒤 전체 빌드로 검증한다.

## 오버라이딩 함수에서 기반 구현 호출

### 질문

기반 클래스와 파생 클래스의 `Initialize()`를 `virtual`과 `override`로 선언하면 파생 함수 내부의 재귀 호출을 해결할 수 있는가?

### 핵심 답변

- `virtual`은 기반 클래스 포인터나 참조를 통한 호출이 실제 객체의 재정의 함수로 연결되게 한다.
- 파생 클래스의 `Initialize()` 안에서 한정 없이 `Initialize(position)`을 호출하면 현재 파생 함수가 다시 호출되므로 무한 재귀가 발생한다.
- 기반 구현을 재사용하려면 가상 함수 여부와 관계없이 `CNotice::Initialize(position)`처럼 클래스 범위를 명시해야 한다.
- `override`는 재정의 의도가 있을 때 컴파일러가 시그니처 일치를 검사하게 하지만 기반 구현을 자동 호출하지는 않는다.

### 프로젝트 적용

현재 호출 경로에는 런타임 다형성이 필요하지 않으므로 `virtual`을 추가하지 않고 `CExit::Initialize()`에서 `CNotice::Initialize(position)`을 명시적으로 호출한다.

## 프러스텀의 구 교차 판정

### 질문

점과 평면 사이의 거리를 계산하는 함수 이름에 왜 `Sphere`가 들어가는가?

### 핵심 답변

- 함수는 전달된 위치와 각 프러스텀 평면 사이의 부호 있는 거리를 계산한다.
- 위치를 구의 중심, 두 번째 거리 값을 반지름으로 보고 `평면 거리 < -반지름`이면 구가 해당 평면 바깥에 완전히 놓였다고 판정한다.
- 어느 평면에서도 완전히 배제되지 않으면 구가 프러스텀 내부에 있거나 경계와 교차하므로 `IntersectsSphere`가 실제 반환 의미에 가깝다.
- 구가 완전히 내부에 있는지만 검사하는 `ContainsSphere`와는 판정 조건이 다르다.
- 반지름을 `0`으로 전달하면 같은 식을 점 포함 판정으로 사용할 수 있다.

### 프로젝트 적용

`CFrustum::IsInFrustum(position, distance)`를 `IntersectsSphere(center, radius)` 의미로 해석하고, 함수 이름은 `IntersectsSphere`로 변경한다. 매개변수 이름 통일은 별도 단계에서 진행한다.

## 대표 타입이 없는 자유 함수 모듈의 파일명

### 질문

`main.cpp`처럼 관례적인 진입점이 아닌 `maze_function.h/.cpp` 같은 자유 함수 모듈은 현재 파일명 규칙의 예외로 두어도 되는가?

### 핵심 답변

- 대표 클래스가 없는 파일도 이름만 보고 책임을 파악할 수 있어야 한다.
- `function`, `util`, `helper`처럼 범위가 넓은 이름보다 모듈이 다루는 도메인이나 역할을 사용한다.
- 프로젝트 파일 규칙이 `PascalCase`라면 자유 함수 모듈도 같은 표기법을 적용한다.
- 언어나 도구가 요구하는 `main.cpp` 같은 관례적 진입점은 예외로 유지할 수 있다.

### 프로젝트 적용

미로 벽, 표지판과 출구 생성을 담당하는 `maze_function.h/.cpp`를 `MazeGenerator.h/.cpp`로 변경한다.

## private 멤버 변수의 `m_` 접두사

### 질문

private 멤버 변수를 `m_` + `camelCase`로 작성하는 방식은 실제로 널리 사용되고 선호되는가?

### 핵심 답변

- `m_memberName`은 C++에서 널리 쓰이는 선택지 중 하나지만 언어 전체의 단일 표준이나 압도적인 선호 방식은 아니다.
- 대표적인 대안으로 접미사를 붙이는 `member_name_`, 별도 표식 없는 `memberName` 또는 `MemberName`이 있다.
- `m_`는 자료형이 아니라 멤버라는 범위를 표시하므로 `p`, `dw` 같은 타입 기반 헝가리안 표기와 목적이 다르다.
- 장점은 매개변수와 지역 변수에서 멤버를 즉시 구분하고 이름 충돌을 줄이는 것이며, 단점은 접두사로 인한 시각적 잡음이다.
- 어떤 형식을 택하든 프로젝트 전체 일관성과 도구 자동 검사가 특정 형식 자체보다 중요하다.

### 프로젝트 적용

기존 코드에 `m_` 계열 이름이 많고 전역 변수에는 `g_` 범위 표식을 사용하므로, private 멤버는 `m_` + `camelCase`로 통일한다.

### 질문

다크 테마에서 C++ 매개변수의 진한 파란색이 잘 보이지 않을 때 어떤 색상이 적절한가?
